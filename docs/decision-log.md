# NexusFlow / 决策记录

## D-001 先做单场景闭环，不先做通用平台
- 结论：先验证闭环，再扩展平台化能力。
- 原因：冷启动成本、采集负担、误判风险都过高。

## D-002 数据层与能力层分离
- 结论：SQLite 存工作数据；文件系统存能力资产。
- 原因：降低耦合，便于 Git 管理与后续演化。

## D-003 MVP 先人工识别，后自动识别
- 结论：先手动日志、人工标注、周回顾、规则筛选。
- 原因：可解释、低成本、适合冷启动。

## D-004 MVP 只做 1 个示范能力资产
- 结论：不并行孵化多个能力。
- 原因：降低范围蔓延与维护负担。

## D-005 OpenClaw 负责交互与编排，脚本负责数据处理
- 结论：统计、清洗、聚合等重逻辑后置到脚本层。
- 原因：避免把复杂处理逻辑塞进 Skill 或 prompt。

## D-006 当前阶段不做的事项
- 通用自动化平台
- 多用户/团队化
- 复杂 ML 识别
- 向量数据库
- 多 Skill 并发上线
- 自动更新与复杂版本治理

## D-007 文档优先于继续扩实现
- 结论：先固化设计、roadmap、计划与决策记录。
- 原因：防止 session 上下文被压缩后丢失关键判断。

## D-008 本次 ClawTeam strategy-room 以人工方式收口
- 结论：team 最终由人工干预完成收口。
- 原因：strategy-lead 未继续消费 inbox，leader 任务长期 pending。
- 备注：不影响核心评审结论有效性，但说明后续如果继续用 ClawTeam，要关注 leader 收尾机制与 idle/finish 行为。
