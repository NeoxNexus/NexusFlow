# 技术方案评审 Prompt

## 适用场景

当需要对一个新系统、新工作流、新功能或新工具方案进行技术评审，并输出可执行的落地建议时使用。

适合场景：
- 新项目启动
- 工作流设计
- 自动化方案评审
- 技术选型与架构收口
- MVP 边界判断
- 阶段性执行顺序设计

不适合场景：
- 纯信息查询
- 很小的实现细节确认
- 已经范围极清晰、只需要直接编码的任务

---

## 使用目标

输出一份“可决策、可执行、可收口”的方案评审结果，而不是泛泛而谈。

默认目标：
1. 明确真实目标
2. 识别核心矛盾
3. 判断推荐路线
4. 给出 MVP 边界
5. 给出阶段划分与执行顺序
6. 明确风险与不建议做的事项
7. 给出下一步建议

---

## 推荐输入

至少提供以下内容：
- 要评审的对象是什么
- 想解决的问题是什么
- 当前背景与约束是什么
- 是否已有已有方案/现状
- 希望输出偏向：架构 / MVP / roadmap / 风险 / 执行顺序

可选补充：
- 时间限制
- 资源限制
- 当前工具栈
- 用户偏好

## Prompt 模板

请对以下主题做一次技术方案评审，并输出一个决策导向的建议。

### 评审对象
{{评审对象}}

### 背景
{{背景}}

### 目标
{{目标}}

### 约束
{{约束}}

### 当前现状 / 已知方案
{{当前现状}}

### 期望输出重点
{{输出重点}}

请按以下原则进行评审：
- 先判断真正目标，再评估方案
- 区分短期 MVP 与长期演进
- 优先给推荐路线，不只罗列选项
- 不把未验证内容说成事实
- 优先输出最小正确、可落地方案

请按以下结构输出：
1. 结论
2. 问题归因 / 核心矛盾
3. 推荐方案
4. 为什么这样做
5. MVP 范围
6. 阶段划分与执行顺序
7. 风险与边界
8. 不建议做的事项
9. 下一步

## 最小示例

### 评审对象
为一个基于 OpenClaw 的个人工作效率提升系统做技术方案评审。

### 背景
希望收集日常工作记录，识别重复任务，并沉淀为可复用能力。

### 目标
确定推荐架构、MVP 范围和实施顺序。

### 约束
不希望一开始做成大而全平台；优先最小可行方案。

### 当前现状 / 已知方案
已有基础设计、SQLite、模板和文档骨架。

### 期望输出重点
推荐架构、MVP 边界、阶段执行顺序、风险与不建议事项。
