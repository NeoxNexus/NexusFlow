# NexusFlow / 架构设计细稿

## 1. 设计目标

本系统用于构建一个围绕 OpenClaw 的个人工作效率提升闭环：

1. 收集日常工作记录
2. 识别高频重复任务
3. 评估哪些任务适合沉淀为能力资产
4. 将高价值流程固化为 Skill / Prompt / Script / Checklist
5. 基于反馈持续优化、下线或重构能力资产

本系统当前阶段的目标不是建设一个通用自动化平台，而是先验证最小闭环是否成立。

---

## 2. 核心设计原则

### 2.1 先闭环，再自动化
先证明：
- 工作记录能稳定积累
- 重复任务能被识别
- 候选任务能被正确分类
- 至少 1 个能力资产确实有价值

再考虑更强自动化和通用化。

### 2.2 数据层与能力层解耦
- 工作记录、候选池、反馈等放 SQLite
- Skills / Prompts / Scripts / Checklists 放文件系统

不要把数据 schema 和 Skill 定义直接耦合。

### 2.3 先人工识别，后自动识别
MVP 阶段优先采用：
- 手动日志
- 规则筛选
- 周回顾
- 人工评审

自动识别（embedding、聚类、频率告警）后置。

### 2.4 先单场景，再通用化
第一阶段只允许围绕 1 个高价值场景打穿闭环。

## 3. 分层架构

```text
[用户日常工作]
      |
      v
+----------------------+
| 1. 采集层            |
| - OpenClaw 会话输入   |
| - 手动工作日志        |
| - 可选脚本钩子        |
+----------------------+
      |
      v
+----------------------+
| 2. 数据/分析层        |
| - SQLite 事件记录     |
| - 周统计              |
| - 候选识别            |
| - 候选评分            |
+----------------------+
      |
      v
+----------------------+
| 3. 能力沉淀层         |
| - Skills              |
| - Prompts             |
| - Scripts             |

+----------------------+
      |
      v
+----------------------+
| 4. 反馈治理层         |
| - 执行反馈            |
| - ROI 评估            |
| - 下线/重构建议       |
+----------------------+
```

---

## 4. 层级职责

### 4.1 采集层
职责：记录工作事件与最少必要上下文。

输入源优先级：
1. OpenClaw 会话输入
2. 手动日志入口
3. 可选脚本钩子（后置）

采集层不负责：
- 自动识别
- 候选推荐
- 技能生成

### 4.2 数据/分析层
职责：
- 存储原始工作事件
- 保存人工标注
- 生成候选池
- 生成周统计
- 支持优先级判断

原则：原始事件与分析结果分开存储。

### 4.3 能力沉淀层
职责：
- 保存能力资产定义
- 管理 Skill / Prompt / Script / Checklist
- 作为沉淀与复用层

原则：
- 能力层不要依赖数据库内部实现细节
- 通过稳定输入输出与数据层协作

### 4.4 反馈治理层
职责：
- 保存能力执行记录
- 记录评分与反馈
- 评估 ROI
- 推动下线、重构、合并

---

## 5. OpenClaw 与外部脚本的边界

### 5.1 OpenClaw 负责
- 接收任务输入
- 引导补充日志
- 调用已有能力资产
- 输出结构化结果
- 协助评审与总结

### 5.2 外部脚本负责
- SQLite 写入与查询
- 周统计
- 候选打分
- 批量清洗与导出
- 固定数据处理逻辑

### 5.3 不推荐的做法
- 不要把大量统计逻辑塞进 Skill
- 不要把复杂评分逻辑塞进 prompt
- 不要让 Skill 强绑定数据库内部 schema

---

## 6. 数据与能力资产组织

### 6.1 数据资产
- workflow_events
- workflow_annotations
- workflow_candidates
- capability_executions
- weekly_reviews

### 6.2 能力资产
- Skills
- Prompts
- Scripts
- Checklists

两类资产分别维护、分别演化。

---

## 7. 能力形态判定原则

### 7.1 适合 Skill
- 触发条件清晰
- 输入输出稳定
- 执行路径明确
- 可重复调用

### 7.2 适合 Prompt
- 更偏思考框架
- 输出结构相对稳定
- 执行步骤不固定

### 7.3 适合 Script
- 逻辑确定性强
- 核心价值在计算、清洗、统计、导出

### 7.4 适合 Checklist
- 需要人工判断
- 更关注避免遗漏
- 不适合全自动

---

## 8. MVP 边界

MVP 仅包含：
- 轻量被动采集
- 手动日志入口
- SQLite 最小 schema
- 周统计 Top10
- 1 个示范能力资产
- 简单反馈机制（建议 5 星评分）

MVP 不包含：
- 复杂 ML 识别
- 向量数据库
- 多 Skill 并发上线
- 多用户/团队协作平台化
- 自动更新/复杂版本治理

---

## 9. 待后续确认项

后续需要在实现阶段补充确认：
1. 是否增加命令执行钩子或文件变更钩子
2. weekly review 的具体生成方式（脚本生成还是 Agent 辅助）
3. 候选评分公式是否需要量化到固定分值
4. 第一个示范能力资产具体选择哪个场景

---

## 10. 当前结论

当前系统应采用：
- 轻采集
- 弱耦合
- 先人工后自动
- Skill 与数据层分离
- 单场景闭环优先

这是当前最小正确、最容易验证、也最便于后续扩展的方案。
